# Phase 2 핀테크 계획서 — 커피 응원 + 내기 (v0.2, 2026-06-11)

> v0.2: 내기도 현금이 아니라 **실물 교환권(기프티콘)** 기반으로 전환 — "커피 내기" 구조.
> 예치 이슈 소멸, 판돈이 일시오락 수준으로 자연 고정, 응원 한 잔과 인프라 통일.

> 작성 배경: Phase 2 의 "선물하기 / 내기하기" 를 실돈 기능으로 올리기 위한 단계별 계획.
> **대원칙: 두하다는 돈과 금융정보를 직접 만지지 않는다.** 이 원칙 하나로
> "해킹으로 회원 금융정보 유출 / 계좌이체 사고" 시나리오를 구조적으로 차단한다.
> 본 문서의 법률·심사 관련 내용은 가이드라인 기반 정리이며, 실돈 오픈 전 전문 자문으로 최종 확인한다.

| # | 원칙 | 막아주는 사고 |
|---|------|--------------|

## 1. 기능 ① — "응원 한 잔" (구 선물하기)

### 1.1 컨셉
- 말 응원(💛 4종 칩)의 윗단계: **실제 커피 한 잔을 보내는 응원**.
- 용어는 "선물하기" 대신 **"응원 한 잔"** (후보: 커피 응원 / 한 잔 쏘기 — 결정 필요 ①).
- v2.5 사상과 정합: 좋아요 인플레이션이 아니라 "목격받기"의 물성화. 도전 인연이 내 여정에 커피 한 잔을 놓고 가는 것.

### 1.2 상품 구성 (2~3종 고정, 임의 금액 입력 금지)
| 티어 | 예시 | 소비자가 | 비고 |
|------|------|---------|------|
| 🧃 음료수 | 편의점 음료 교환권 | 3,000원 안팎 | 가볍게 |
| ☕ 커피 | 메가/컴포즈/스벅 아메리카노 | 3,500~4,500원 | 기본 |
| 🍰 커피+디저트 | 케이크 세트 | 8,000~10,000원 | 3종으로 갈 경우 (결정 필요 ②) |

- 임의 금액 입력을 막는 이유: 자금세탁·증여 우회 통로 차단 + 심사 단순화 + UI 단순.
- 1일/1인당 발신 한도(예: 일 3건) 를 처음부터 둔다 — 어뷰징·결제사고 폭발 반경 제한.

### 1.3 돈의 흐름 (에스크로 아님 — 단건 매입 차익 모델)
```
보내는 사람 ──(PG 결제창: 카드/간편결제)──> 토스페이먼츠 ──(정산 D+2~7)──> 두하다 사업자 계좌
두하다 ──(기프티콘 B2B API 매입, 할인가)──> 받는 사람 휴대폰으로 교환권 발송
```
- **마진 = 소비자가 − PG 수수료(약 3%+VAT) − 기프티콘 B2B 매입가** (B2B 할인 통상 3~10%).
- 기프티콘 B2B 공급사 후보: 기프티쇼 비즈(KT alpha), 쿠프마케팅, 즐거운(주) — Stage 3 에서 비교 견적.
- 에스크로 불필요: 돈을 "보관"하는 게 아니라 "상품을 사서 배달"하는 단건 거래라서. 보관이 없으면 보관 사고도 없다.

### 1.4 앱스토어 심사 관점 (이 설계를 택한 이유)
- Apple 가이드라인 3.1.3(e): **앱 밖에서 소비되는 실물 상품/서비스는 인앱결제 대상이 아님** → 카페에서 교환하는 기프티콘은 PG 결제 허용 (카카오 선물하기와 동일 구조).
- 반면 "현금 응원 + 앱 마진" 구조였다면: 3.2.1(vii) 상 개인 간 금전 선물은 100% 전달일 때만 외부결제 허용 → 마진 떼는 순간 IAP 30% 강제 → 사업성 붕괴. **그래서 기프티콘 구조가 유일한 현실 답.**
- Google Play 도 실물 상품은 인앱결제 의무 면제 — 동일 논리.

### 1.5 상태머신 (자동 테스트 의무 영역)
```
created → paid → issued → delivered → redeemed
   │        │       │
   │        │       └─ issue_failed → auto_refund (기프티콘 발급 실패 시 전액 자동 환불)
   │        └─ pay_failed (종결)
   └─ canceled (결제창 이탈)
```
- 핵심 인변량: **paid 인데 issued/refund 둘 다 아닌 상태가 N분 이상 지속되면 알람** (돈만 받고 상품 못 준 상태 = 최악).
- 환불 정책: 받는 사람 미수령 N일 경과 시 자동 환불(또는 재발송) — 약관에 명시.

### 1.6 클라이언트가 절대 결정하지 못하는 것 (서버 검증 의무)
- 금액·상품ID: Edge Function 이 PG 승인 API 응답의 금액과 주문의 금액을 **서버에서 대조**. 불일치 → 즉시 결제 취소.
- 수신자: 같은 챌린지 멤버인지 서버에서 검증 (도전 인연 = 같은 방 멤버 원칙과 동일 잣대).
- 미성년자: 만 19세 미만 수신/발신 정책 결정 필요 ③ (기획서 8.3 연장 — 베타는 차단 권장).

## 2. 기능 ② — "커피 내기" (구 내기하기 · 실물 교환권 기반, 게이트 통과 전 오픈 금지)

### 2.1 구조 — 현금 베팅이 아니라 "기프티콘 선주문 + 조건부 수령"
```
내기 시작: 참여자 전원이 같은 티어 상품을 각자 선주문 결제 (예: ☕ 커피 1잔씩)
            └ 이 시점에 매출 확정 — 일반 이커머스 선주문과 동일, 돈을 "보관"하지 않음
도전 기간: 아무것도 안 움직임 (발급 보류 상태)
정산 시점: 완주    → 자기가 주문한 기프티콘을 자기가 수령 (사실상 전액 환급과 동일)
            미완주 → 자기 기프티콘이 규칙에 따라 처리 (완주자 분배 / 기부 — 결정 필요 ④)
            전원 미완주·방 폭파·개설자 포기 → 전원 전액 환불
```
- **"남의 돈을 딴다" ❌ → "내 커피를 지킨다" ⭕** — 챌린저스의 환급 프레임 + 기프티콘 결합.
- 결과는 우연이 아니라 **본인 노력(자기 완주 여부)** 으로 결정 — 도박 구성요건(우연성)과 거리 확보.
- 판돈이 커피값(3천~1만원)으로 자연 고정 — 형법 246조 단서 **"일시오락 정도" 예외** 방어선에 부합. "커피 내기"는 문화적으로도 일상 오락 표현이라 심사·사용자 인식 양쪽에 유리.
- 타인 완주 여부에 거는 구조(관전 베팅)는 우연성 베팅이 되므로 **영구 금지**.

### 2.2 기프티콘 전환이 해결하는 것 / 못 하는 것 (정직한 정리)
| | 내용 |
|---|------|
| ⭕ 예치 소멸 | 시작 시점에 상품 선주문으로 매출 확정 — "보증금 보관" 자체가 없어 전금법 예치 이슈 회피. PG 정산보류 질의(구 결정 ⑤) 불필요 |
| ⭕ 페이아웃 소멸 | 정산 = 현금 이체가 아니라 기프티콘 발급 API 호출. 원칙 2 유지 |
| ⭕ 인프라 통일 | 응원 한 잔과 같은 상품 카탈로그·B2B 발급 파이프라인·상태머신 기반 — Stage 1 테스트 코드 재사용 |
| ⭕ IAP 일관성 | 실물 교환권이라 인앱결제 면제 논리가 응원 한 잔과 동일 |
| ⚠️ 도박죄 완전 면제는 아님 | 기프티콘도 법적으로 "재산상 이익"(중고 현금화 가능). 미완주분 분배 규칙은 여전히 자문 안건 — 다만 자문 범위가 "예치+송금+도박성"에서 "분배 규칙+약관"으로 크게 축소 |
| ⚠️ 청약철회 충돌 (신규) | 전자상거래법 청약철회 7일은 강행규정 — "도전 중간 환불 요구"와 내기 구조의 충돌을 약관으로 푸는 방법이 자문 안건에 추가 |
| ⚠️ 유효기간 운영 (신규) | 기프티콘 유효기간(통상 30~93일) 때문에 발급은 **정산 시점**에 한다 (시작 시점 선발급 금지). B2B 계약 시 발급 시점 유연성 확인 |

### 2.3 변호사 검토에 대한 답
- **앱 심사 자체는 변호사 검토를 요구하지 않는다.** Apple/Google 심사관은 법률 의견서를 안 본다.
- 그러나 ① 내기류 기능은 심사에서 도박 카테고리로 오인될 수 있고 ② 심사 통과 ≠ 합법 — 도박죄·사행행위규제법 판단은 별개.
- **권장: 커피 내기 오픈 전, 구조 확정 시점에 1회 법률 자문** (단건 수십만 원대 — 폭탄 리스크 대비 가장 싼 보험). 베타·1차 정식 출시에는 내기를 빼고 출시.
- 자문 안건 (좁혀진 3개): ⓐ 미완주분 분배 규칙의 사행성 ⓑ 청약철회와 내기 약관 설계 ⓒ 일시오락 예외 적용 범위(금액 상한). 준비물: 본 문서 + 돈 흐름도 + 약관 초안.

### 2.4 내기 상태머신 (자동 테스트 의무 영역 — 구현은 게이트 통과 후)
```
proposed → funded(전원 선주문 결제 완료) → active → settled(완주판정 → 수령/처리 계산)
                                              │         ├→ issued (각자 몫 기프티콘 발급)
                                              │         └→ issue_failed → auto_refund
                                              └→ all_refund (방 폭파·개설자 포기·전원 미완주 등)
```
- 완주 판정은 기존 `lib/stats.ts` 단일 소스 재사용 (KST·frequency 규칙 그대로).
- 핵심 인변량: funded 인원수 × 단가 = 주문 합계 (정산 대사), settled 후 모든 참가자의 몫 합계 = 전체 주문 수량 (한 장도 증발·복제되지 않음).

- [ ] 첫 1주 매일 정산 대사(우리 DB 주문 합계 = PG 정산 내역) 수동 확인

### Stage 5 — 커피 내기 (별도 게이트)
- [ ] 변호사 자문 통과 (안건 ⓐⓑⓒ — 2.3절) → Stage 1 정산 로직 + 응원 한 잔 발급 파이프라인 재사용으로 연결 → 별도 심사 노트 준비

## 6. 이 계획이 원래 두려움에 답하는 방식

| 두려움 | 답 |
|--------|-----|
| 해킹으로 회원 금융정보 유출 | 원칙 1 — 우리 DB 에 카드·계좌 정보가 존재하지 않음. 털려도 나올 게 주문번호뿐 |
| 계좌이체로 돈이 잘못 나감 | 원칙 2 — 우리가 이체를 안 함. 나가는 건 기프티콘 발급 API 호출 1건, 실패하면 자동 환불 |
| 정산 계산 실수 | 상태머신 + 전수 단위 테스트 + 매일 정산 대사 (Stage 4) |
| 법/심사 폭탄 | 응원·내기 모두 실물 교환권 구조(IAP 면제·송금업·예치 회피) + 내기 판돈은 커피값 = 일시오락 수준 + 법률 자문 게이트 |
| 실수해도 망하지 않게 | 실돈은 Stage 3 에야 등장, 그때도 본인 카드 + 한도 1만원부터 |
