# Phase 2 핀테크 계획서 — 응원 한잔 + 내기 한잔 (v0.4, 2026-06-11)

> v0.2: 내기도 현금이 아니라 **실물 교환권(기프티콘)** 기반으로 전환 — 예치 이슈 소멸, 응원과 인프라 통일.
> v0.3: 사용자 결정 반영 — 명칭 **응원 한잔 / 내기 한잔**, 티어 확정(응원 2종·내기 3종),
> 미성년자 내기 금지(방 단위), **미완주분 = 기부 확정 + 기부 허브 신설**.
> 기부 처분 확정으로 "남의 것을 따는" 구조가 완전히 사라져 도박 구성요건이 사실상 소멸 (2.1절).
> v0.4: 미성년자 확인 = 자기신고 → **휴대폰 본인인증** 으로 격상 (돈 기능 첫 사용 시 1회 — 2.1-2절),
> 기부 집행 = **물품(기프티콘) 기부 우선** + 공제·마진 공시 원칙 (4.3절).
> v0.5 (2026-06-12): **나와의 내기** (나홀로·응원받기 방 — 완주=본전, 실패 인정=기부, 2.1-3절) +
> 기부 시 **운영비 공제 공시 문구** (4.3절). 나와의 내기는 상호 득실이 없어 내기 1차 오픈 후보.

> 작성 배경: Phase 2 의 "선물하기 / 내기하기" 를 실돈 기능으로 올리기 위한 단계별 계획.
> **대원칙: 두하다는 돈과 금융정보를 직접 만지지 않는다.** 이 원칙 하나로
> "해킹으로 회원 금융정보 유출 / 계좌이체 사고" 시나리오를 구조적으로 차단한다.
> 본 문서의 법률·심사 관련 내용은 가이드라인 기반 정리이며, 실돈 오픈 전 전문 자문으로 최종 확인한다.

| # | 원칙 | 막아주는 사고 |
|---|------|--------------|

## 1. 기능 ① — "응원 한잔" (구 선물하기) ✅ 명칭 확정

### 1.1 컨셉
- 말 응원(💛 4종 칩)의 윗단계: **실제 한 잔을 보내는 응원**.
- v2.5 사상과 정합: 좋아요 인플레이션이 아니라 "목격받기"의 물성화. 도전 인연이 내 여정에 커피 한 잔을 놓고 가는 것.
- 받는 사람은 수령 대신 **"기부로 돌리기"** 선택 가능 (4절 기부 허브와 연결).

### 1.2 상품 구성 ✅ 2종 확정 (임의 금액 입력 금지)
| 티어 | 예시 | 소비자가 |
|------|------|---------|
| ☕ 한잔 | 커피 교환권 | 5,000원 |
| 🍰 든든한 한잔 | 커피+디저트 세트 | 10,000원 |

- 구체 상품(브랜드)은 Stage 3 B2B 카탈로그 보고 매칭. 금액 축은 5천/1만 고정.
- 임의 금액 입력을 막는 이유: 자금세탁·증여 우회 통로 차단 + 심사 단순화 + UI 단순.
- 1일/1인당 발신 한도(예: 일 3건) 를 처음부터 둔다 — 어뷰징·결제사고 폭발 반경 제한.

### 1.3 돈의 흐름 (에스크로 아님 — 단건 매입 차익 모델)
```
보내는 사람 ──(PG 결제창: 카드/간편결제)──> 토스페이먼츠 ──(정산 D+2~7)──> 두하다 사업자 계좌
두하다 ──(기프티콘 B2B API 매입, 할인가)──> 받는 사람 휴대폰으로 교환권 발송
```
- **마진 = 소비자가 − PG 수수료(약 3%+VAT) − 기프티콘 B2B 매입가** (B2B 할인 통상 3~10%).
- 기프티콘 B2B 공급사 후보: 기프티쇼 비즈(KT alpha), 쿠프마케팅, 즐거운(주) — Stage 3 에서 비교 견적.
- 에스크로 불필요: 돈을 "보관"하는 게 아니라 "상품을 사서 배달"하는 단건 거래라서. 보관이 없으면 보관 사고도 없다.

### 1.4 앱스토어 심사 관점 (이 설계를 택한 이유)
- Apple 가이드라인 3.1.3(e): **앱 밖에서 소비되는 실물 상품/서비스는 인앱결제 대상이 아님** → 카페에서 교환하는 기프티콘은 PG 결제 허용 (카카오 선물하기와 동일 구조).
- 반면 "현금 응원 + 앱 마진" 구조였다면: 3.2.1(vii) 상 개인 간 금전 선물은 100% 전달일 때만 외부결제 허용 → 마진 떼는 순간 IAP 30% 강제 → 사업성 붕괴. **그래서 기프티콘 구조가 유일한 현실 답.**
- Google Play 도 실물 상품은 인앱결제 의무 면제 — 동일 논리.

### 1.5 상태머신 (자동 테스트 의무 영역)
```
created → paid → issued → delivered → redeemed
   │        │       │
   │        │       └─ issue_failed → auto_refund (기프티콘 발급 실패 시 전액 자동 환불)
   │        └─ pay_failed (종결)
   └─ canceled (결제창 이탈)
```
- 핵심 인변량: **paid 인데 issued/refund 둘 다 아닌 상태가 N분 이상 지속되면 알람** (돈만 받고 상품 못 준 상태 = 최악).
- 환불 정책: 받는 사람 미수령 N일 경과 시 자동 환불(또는 재발송) — 약관에 명시.

### 1.6 클라이언트가 절대 결정하지 못하는 것 (서버 검증 의무)
- 금액·상품ID: Edge Function 이 PG 승인 API 응답의 금액과 주문의 금액을 **서버에서 대조**. 불일치 → 즉시 결제 취소.
- 수신자: 같은 챌린지 멤버인지 서버에서 검증 (도전 인연 = 같은 방 멤버 원칙과 동일 잣대).
- 미성년자: 만 19세 미만 수신/발신 정책 결정 필요 ③ (기획서 8.3 연장 — 베타는 차단 권장).

## 2. 기능 ② — "내기 한잔" ✅ 명칭 확정 (실물 교환권 기반, 게이트 통과 전 오픈 금지)

### 2.1 구조 — "기프티콘 선주문 + 완주하면 내 것, 미완주면 기부"
```
내기 시작: 참여자 전원이 같은 티어 상품을 각자 선주문 결제 (예: ☕ 5,000원 1잔씩)
            └ 이 시점에 매출 확정 — 일반 이커머스 선주문과 동일, 돈을 "보관"하지 않음
도전 기간: 아무것도 안 움직임 (발급 보류 상태)
정산 시점: 완주    → 자기가 주문한 기프티콘을 자기가 수령 (수령 대신 "기부로 돌리기" 선택 가능)
            미완주 → 자기 몫이 **기부 풀로 전환** (4절 기부 허브) ✅ 확정 — 완주자 분배 없음
            전원 미완주·방 폭파·개설자 포기 → 전원 전액 환불
```
- **도박 판단 (v0.3 핵심)**: 기부 처분 확정으로 **참여자 누구도 타인의 재산을 얻지 않는다** —
  완주자는 자기 돈으로 산 자기 상품을 받을 뿐, 미완주분은 제3자(기부처)로. 도박죄의 핵심 구성요건인
  "재물을 걸고 우연한 승부로 **상호 간 득실**을 결정"이 구조적으로 성립하지 않음.
  해외 선례 = stickK (실패 시 기부 약정 commitment device), 국내 = 챌린저스 벌금 일부 기부.
- 결과는 우연이 아니라 **본인 노력(자기 완주 여부)** 으로 결정 — 우연성 요건과도 거리 확보 (이중 방어).
- 타인 완주 여부에 거는 구조(관전 베팅)는 상호 득실+우연성이 모두 살아나므로 **영구 금지**.
- 완주자 분배 옵션은 어떤 형태로도 **재도입 금지** — 재도입하는 순간 위 판단 전체가 무효.

### 2.1-1 티어 ✅ 3종 확정 (5천/1만/2만)
| 티어 | 소비자가 | 도박 영역 판단 |
|------|---------|---------------|
| ☕ 한잔 | 5,000원 | 문제없음 |
| 🍰 든든한 한잔 | 10,000원 | 문제없음 |
| 🎁 거하게 한잔 | 20,000원 | **판단: 가능** — 득실 이전이 없는 구조라 금액은 도박 판단이 아니라 소비자보호(환불) 관점 이슈. 단, 자문 안건 ⓒ에서 2만원 상한 명시 확인 |

- 2만원 판단 근거: "일시오락" 금액 논쟁은 참여자 간 돈이 오갈 때의 이야기. 본 구조는 득실 이전 자체가
  없어 금액이 커져도 도박성이 커지지 않음 (자기 소비 + 실패 시 기부). 잔여 리스크는 도박이 아니라
  **기부금품법**(4.3절)과 **청약철회** 쪽이며, 이는 자문 안건으로 관리.

### 2.1-2 미성년자 내기 금지 ✅ 확정 — 방 단위 차단 + 휴대폰 본인인증 (v0.4 격상)
- **규칙: 챌린지방 멤버(동료 포함) 중 미성년자(또는 미인증자)가 1명이라도 있으면 그 방은 내기 한잔 비활성.**
- **확인 수단 = 휴대폰 본인인증** (자기신고 폐기): PASS/NICE/KCB 본인확인 서비스 — 실명·생년월일·휴대폰번호가
  통신사 기준으로 확보됨. ※ 문자 OTP(번호 확인)와는 다른 서비스 — 본인확인 대행 계약 필요, 건당 수십 원.
- **인증 시점 = 돈 기능 첫 사용 시 1회**: 응원 한잔 보내기 / 내기 만들기 / 내기 참여하기 첫 시도 때
  본인인증 → `users` 에 `birth_date`·`phone`·`verified_at`(+중복 방지용 DI) 저장.
  **무료 핵심 루프(일반 챌린지 만들기·참여)에는 걸지 않는다** — 베타 전환율 보호 + 미성년자도 무료 기능은 자유.
- **미인증 = 미성년자로 간주** (보수적): 인증 안 한 멤버가 있는 방도 내기 비활성.
  내기를 원하는 사용자가 스스로 인증하게 되는 자연 유인 구조.
- 일석삼조: 미성년 차단 + 결제자 실명(분쟁·환불 대응) + 중복 계정 방지(DI).
  법적 보강 이유: **미성년자 결제는 민법상 법정대리인 취소권** 대상 — 차단이 보호 정책이자 사업 방어 (자문 안건 ⓓ).
- 저장 최소화 원칙: CI 원문 등 과잉 수집 금지 — 생년월일·인증시각·DI 정도만. 개인정보처리방침 갱신 필수 (3절).

### 2.1-3 나와의 내기 — 나홀로·응원받기 방 (v0.5 추가, 사용자 제안)
```
나홀로(solo)·응원받기(cheered) 방에서 도전자 본인이 자기 한잔을 선주문 결제
정산: 완주          → 본전 — 자기 기프티콘 수령 (또는 "기부로 돌리기" 선택)
      실패·중도포기 → 기부 (환불 없음 — "실패를 인정하면 기부" 약속)
      시스템 무효(계정 탈퇴 등 예외 상황만) → 환불
```
- **다인 내기와 규칙이 다른 지점 (의도적 분기)**: 다인 내기의 "전원 미완주 → 전원 환불"을
  1인에 적용하면 혼자 실패 = 환불이 되어 내기가 무력화됨. 나와의 내기는 **실패 = 기부 확정** —
  이 긴장감이 기능의 존재 이유 (stickK 커밋먼트 계약의 원형).
- **실패 인정 UX**: 가입 시 약관 동의("실패 시 기부됩니다") + 정산 시점에 "실패를 인정하고
  기부할까요?" 확인 1회. 미응답 시 N일 후 자동 기부 (paid 잔류 방지 — 기한은 약관 명시, 자문 ⓑ 안건).
- **법적 평가**: 참여자 간 득실 이전이 **원천적으로 존재하지 않음** (상대방이 없음) → 사행성
  논점이 사실상 소멸. 다인 내기보다 안전해서 **자문 통과 후 내기 1차 오픈(파일럿) 후보**.
- cheered 방은 도전자(개설자)만 가능 — 응원자들이 지켜보는 가운데의 자기 약속 (목격받기 시너지).
- 구현 메모: `betSettlement.ts` 에 `mode: 'self'` 분기 (Stage 1 순수 로직 선행 구현 + 테스트 완료),
  주문은 기존 `gift_orders` order_type='bet' + 1인 구조 재사용.

### 2.2 기프티콘 전환이 해결하는 것 / 못 하는 것 (정직한 정리)
| | 내용 |
|---|------|
| ⭕ 예치 소멸 | 시작 시점에 상품 선주문으로 매출 확정 — "보증금 보관" 자체가 없어 전금법 예치 이슈 회피. PG 정산보류 질의(구 결정 ⑤) 불필요 |
| ⭕ 페이아웃 소멸 | 정산 = 현금 이체가 아니라 기프티콘 발급 API 호출. 원칙 2 유지 |
| ⭕ 인프라 통일 | 응원 한 잔과 같은 상품 카탈로그·B2B 발급 파이프라인·상태머신 기반 — Stage 1 테스트 코드 재사용 |

  ② "참여하려면 휴대폰 본인인증(성인)이 필요해요" 안내 — 초대받은 사람이 합류 전에 조건을 알게.

### 2.3 변호사 검토에 대한 답
- **앱 심사 자체는 변호사 검토를 요구하지 않는다.** Apple/Google 심사관은 법률 의견서를 안 본다.
- 그러나 ① 내기류 기능은 심사에서 도박 카테고리로 오인될 수 있고 ② 심사 통과 ≠ 합법 — 도박죄·사행행위규제법 판단은 별개.
- **권장: 내기 한잔 오픈 전, 구조 확정 시점에 1회 법률 자문** (단건 수십만 원대 — 폭탄 리스크 대비 가장 싼 보험). 베타·1차 정식 출시에는 내기를 빼고 출시.
- 자문 안건 (v0.5 갱신, 4개): ⓐ **기부 전환 구조와 기부금품법** — 모집 등록 의무 여부, 집행 구조(4.3절 3안 중 택일) + **운영비 공제율 상한·공시 방식** ⓑ 청약철회(강행규정)와 내기 약관 설계 + **나와의 내기 "실패 인정 → 기부" 약정·미응답 자동 기부 기한** ⓒ 득실 이전 없는 구조에서 2만원 상한 적정성 + **나와의 내기(상대 없음) 우선 오픈 가능성** ⓓ 휴대폰 본인인증 기반 미성년 차단의 충분성 + 미성년자 결제 취소권 대응. 준비물: 본 문서 + 돈 흐름도 + 약관 초안.

### 2.4 내기 상태머신 (자동 테스트 의무 영역 — 구현은 게이트 통과 후)
```
proposed → funded(전원 선주문 결제 완료) → active → settled(완주판정 → 수령/기부 계산)
                                              │         ├→ issued   (완주자 몫 기프티콘 발급)
                                              │         ├→ donated  (미완주분 + 완주자 기부 선택분 → 기부 풀)
                                              │         └→ issue_failed → auto_refund
                                              └→ all_refund (방 폭파·개설자 포기·전원 미완주 등)
```
- 완주 판정은 기존 `lib/stats.ts` 단일 소스 재사용 (KST·frequency 규칙 그대로).
- 핵심 인변량: funded 인원수 × 단가 = 주문 합계 (정산 대사), settled 후 **issued + donated + refunded 합계 = 전체 주문 수량** (한 장도 증발·복제되지 않음).

## 7. 이 계획이 원래 두려움에 답하는 방식

| 두려움 | 답 |
|--------|-----|
| 해킹으로 회원 금융정보 유출 | 원칙 1 — 우리 DB 에 카드·계좌 정보가 존재하지 않음. 털려도 나올 게 주문번호뿐 |
| 계좌이체로 돈이 잘못 나감 | 원칙 2 — 우리가 이체를 안 함. 나가는 건 기프티콘 발급 API 호출 1건, 실패하면 자동 환불 |
| 정산 계산 실수 | 상태머신 + 전수 단위 테스트 + 매일 정산 대사 (Stage 4) |
| 법/심사 폭탄 | 응원·내기 모두 실물 교환권 구조(IAP 면제·송금업·예치 회피) + 미완주분 기부로 참여자 간 득실 소멸(도박 구성요건 차단) + 법률 자문 게이트 |
| 실수해도 망하지 않게 | 실돈은 Stage 3 에야 등장, 그때도 본인 카드 + 한도 1만원부터 |
