# gemini.md — Gemini AI 작업 규칙

이 파일은 Gemini AI가 이 저장소에서 작업할 때 **반드시 준수해야 하는 개발 지침**이다.

---

## 1. 핵심 개발 및 빌드 원칙

### 1.1 선 허가 후 빌드/배포 원칙 (EAS Build Control)
*   **EAS 빌드 및 배포 금지**: 제미니가 임의로 `eas build` 또는 `eas submit` 명령을 수행하거나 백그라운드 태스크로 실행하지 않는다.
*   **사전 허가 필수**: 빌드 및 배포는 한 번 실행할 때 1시간 이상 소요되므로, 코드 수정이 완료된 후 **사용자의 명시적인 허락(승인)을 구한 후에만 빌드를 제안 및 진행**한다.

### 1.2 컨텍스트 관리 및 새 창 전환 규칙
*   **새 창 전환 타이밍**: 대화가 25~30회 내외로 누적되거나 대규모 빌드 로그/코드 변경이 발생하면 추론 성능 저하 및 맥락 누락이 발생할 수 있다.
*   **세션 이전 준비**: 로그인 테스트가 끝나거나 중대 기능 개발을 시작하기 직전, `walkthrough.md`와 `gemini.md`를 최신화한 후 새 대화 창으로 넘어가기를 사용자에게 권장한다.

### 1.3 안전 영역 (Safe Area) 화면 대응
*   **헤더 겹침 방지**: 모바일 기기의 상단 노치, 상태 바(Status Bar) 영역과 앱 내 UI 헤더(로고, 건너뛰기 버튼 등)가 겹치지 않도록 방지한다.
*   **인셋(Insets) 의무 적용**: 헤더나 최상단 컴포넌트 개발 시 `react-native-safe-area-context`의 `useSafeAreaInsets`를 사용해 상단 패딩/마진을 유동적으로 부여한다.

### 1.4 환경 변수 보안 및 배포 관리
*   **비밀키 보호**: Supabase Service Role Key 등 민감한 API Key는 코드에 노출하지 않으며 `.env` 파일에 보관한다. `.env`는 `.gitignore`를 통해 Git 저장소에 커밋되지 않도록 보호한다.
*   **EAS 빌드 시 주입**: local `.env` 설정이 EAS 빌드 서버로 올라가도록 빌드 제외 항목은 `mobile/.easignore`에서 정밀 제어하되, 절대 원격 Git에 시크릿을 직접 푸시하지 않는다.

## 2. 코드 및 협업 규칙 (CLAUDE.md 대원칙 준수)

### 2.0 Human Readable 원칙 (대전제)
*   모든 코드는 훗날 1인 개발자가 혼자 읽고 유지보수할 수 있어야 한다.
*   변수명·함수명은 역할이 명확히 드러나게 짓고, 한국어 주석을 적극 사용한다.
*   의미 없는 축약어 대신 직관적인 이름을 쓰고, 새 기능 블록 상단에 `// 🚀 기능명: 설명` 주석을 추가한다.

### 2.1 코드 부풀리기 금지 (최우선)
*   **요청한 것만 만든다.** 요구사항 외의 불필요한 기능, 임의의 리팩터링, 추측성 설정 옵션을 추가하지 않는다.
*   불필요한 종속성(라이브러리)을 늘리지 않으며, 추가가 필요한 경우 반드시 사전에 확인받는다.

### 2.2 코드 보호 & Surgical Edit (수술식 편집 의무화)
*   **수술식 편집 적용:** 토큰 소모를 최소화하고 안전하게 코드를 보호하기 위해 파일 전체를 덮어쓰는 `Write` 도구의 사용을 최대한 자제한다. 코드 수정 시에는 오직 변경할 영역만 `replace_file_content` 또는 `multi_replace_file_content` 도구로 정밀하고 좁게 교체한다. (단, 신규 파일 생성 시에만 `Write` 허용)
*   **기존 코드 보호:** 요구사항과 직접 연관이 없는 기존 코드는 건드리지 않는다. 디자인 토큰(`tokens.ts`)도 함부로 수정하지 않는다.
*   **200라인 규칙:** 한 파일은 200라인 이하로 유지하는 것을 지향하며, 복잡도가 늘어나면 컴포넌트 분리를 적극 고려한다.

### 2.3 선 보고 후 실행 (큰 수정) 및 제안 확인 의무
*   구조 변경, 라이브러리 추가, DB 스키마 수정 등 **큰 작업**은 진행 전 **AS-IS → TO-BE**를 설명하고 승인을 얻는다.
*   **제안 확인 의무**: AI 측의 제안이나 임의 판단으로 인한 추가 수정(예: 개수 확장 등)은 독단적으로 즉시 실행하지 않으며, **사전에 사용자에게 제안 내용을 명확히 이야기하고 확인(승인)을 구한 후에만 진행**한다.
*   오타 수정이나 아주 미세한 스타일 보정 등 **작은 수정**은 바로 진행하고 한 줄 보고로 갈음한다.

## 3. 기능 개발 및 검증 규칙

### 3.1 자동 테스트 의무 영역
다음 영역은 사고 시 리스크가 크므로 변경 사항 발생 시 `__tests__/` 하위에 테스트 코드를 함께 작성/수정한다:
1.  **결제 로직** (토스페이먼츠 등)
2.  **AI 콘텐츠 검수** (Claude API 호출 래퍼, 검증)
3.  **인증/권한** (OAuth 콜백, RLS 정책)
4.  **박제 자산화** (완주 인증서 자동 생성)
5.  **유배지/보석금** (페널티 상태 머신)
6.  **도전 인연 ×횟수** (중복 방지, 관계 등급)

### 3.2 수동 검증 영역
*   UI/UX, 네비게이션, 단순 레이아웃은 실기기(Expo Go 등)에서 직접 작동 확인하는 것으로 충분하며 자동 테스트 작성을 강제하지 않는다.

### 3.3 핵심 Do:하다 서비스 원칙
*   **비교 압박 금지**: 거대 숫자 노출 금지 (100+ 이상은 `99+`로 약화), 순위 랭킹 금지.
*   **친구 신청 금지**: "친구", "팔로우" 대신 "도전 인연", "동료"로 표현하며, 관계는 챌린지 참여 시 자동 생성된다.
*   **박제 영구 보존**: 사용자가 명시적으로 삭제하지 않는 한 완주 박제 데이터는 만료/자동 삭제되지 않는다.
*   **콘텐츠 검수 필수**: AI 검수를 건너뛰는 예외 코드를 작성하지 않는다.

## 4. 자주 사용하는 명령

```bash
# 개발 시작
npx expo start

# EAS 빌드 (반드시 사용자의 빌드 승인 후 실행)
eas build --profile development     # 개발 빌드
eas build --profile preview         # 베타 테스트 빌드
eas build --profile production      # 출시 빌드

# Supabase 타입 생성
supabase gen types typescript --project-id <id> > types/database.ts

# 테스트 실행
npm test
```
